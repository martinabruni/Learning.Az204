# Troubleshoot App Performance by Using Application Map

> [!source] learn.microsoft.com_en-us_training_modules_monitor-app-performance_6-application-map.pdf

## Overview

[[application-map]] helps spot performance bottlenecks or failure hotspots across all components of a distributed application. Each node on the map represents an application component or its dependencies, showing health KPIs and alert status.

> [!source] learn.microsoft.com_en-us_training_modules_monitor-app-performance_6-application-map.pdf
> "Application Map helps you spot performance bottlenecks or failure hotspots across all components of your distributed application."

## Key Claims

- Clicking a component node drills into detailed diagnostics (Application Insights events).
- If the app uses Azure services, you can drill through to Azure diagnostics (e.g., SQL Database Advisor recommendations).
- Application Map uses the **cloud role name** property to identify components; this can be manually set or overridden.
- Components are found by following **HTTP dependency calls** between servers with the Application Insights SDK installed.

## What Are Components?

- Independently deployable parts of a distributed/microservices application.
- Developers and operations teams have code-level visibility or access to telemetry generated by these components.
- **Different from "observed" external dependencies** (SQL, Event Hubs, etc.) which the team may not control.
- Components can run on any number of server/role/container instances.
- Components can be:
  - Separate Application Insights instrumentation keys (even across different subscriptions), OR
  - Different roles reporting to a single Application Insights instrumentation key.

## Progressive Discovery

- When the Application Map is first loaded, a set of queries discovers all related components.
- A button in the top-left corner shows the number of discovered components.
- **"Update map components"** button refreshes with all components discovered so far (may take a minute for complex apps).
- If all components are roles within a single Application Insights resource, the discovery step is not required.

## Use Case

- Visualize complex topologies with **hundreds of components**.
- Select any component to see related insights, performance triage, and failure triage.
- The map shows full application topology across **multiple levels** of related components.

## Data Points (from illustration in PDF)

- Example app "fabrikamprod" with 1 instance showing 12% failure rate.
- Connected to: Azure Blob (12.5 ms avg, 25% errors, 1.5K calls), SQL (16.3 ms, 5% errors, 3.8K calls), Azure Queue (9 ms, 1.1K calls).
- Availability probe: 25% success rate, 8.3s avg, 517 tests.
- Top failing requests visible in sidebar: GET ServiceTickets/Details (600 failures), GET Employees/Details (200), POST ServiceTickets/Create (195).
